# Provider 访谈质量优化计划

本文档记录 `v0.1.0-alpha.1` 之后需要重点打磨的 OpenAI-compatible Provider 访谈体验。

这里的 Provider 包括 OpenAI、DeepSeek、Moonshot / Kimi、通义千问 Qwen、OpenRouter、本地模型和自定义 OpenAI-compatible 服务。DeepSeek 是当前真实试用中暴露问题最多的样例，但优化方向不应写死到某一个供应商。

## 当前观察

- 有些模型会重复追问，用户已经表达“不懂、没有计划、后续再考虑”后，仍可能继续问类似问题。
- 部分问题偏专业，例如直接询问框架、部署、数据库、模型细节，代码小白很难回答。
- 不同 Provider 的表达风格、JSON 遵循能力和长上下文稳定性不同，不能假设所有模型都按同一种方式回答。
- 用户真正需要的是把模糊想法整理成可交给 Codex / OpenCode 的任务包，不一定需要自己回答所有工程细节。
- 生成的 Markdown 文件是给 Agent 执行的，所以访谈应重点收集业务目标、使用场景、边界、验收标准和不确定项，而不是逼用户做技术决策。

## 访谈原则

- 先识别用户背景：代码小白、会改代码、熟悉开发。
- 对代码小白优先问：
  - 这个工具给谁用
  - 用户现在痛点是什么
  - 第一版必须完成什么
  - 明确不做什么
  - 页面/流程大概长什么样
  - 什么结果算完成
- 专业工程问题应改成可理解的选择题、默认建议或“交给 Agent 推荐”。
- 用户回答“不懂”“后续再考虑”“暂时没有计划”时，记录为待确认，不继续追问同一专业点。
- 技术实现细节应更多放到 `assumptions`、`recommendations`、`unresolvedQuestions`，交给 Agent 后续执行时再确认。
- 任何模型生成的建议都不能伪装成用户确认事实，输出中要区分“用户已确认”和“AI 建议”。

## Provider 适配边界

Provider 适配应该只处理协议与稳定性差异：

- Base URL、模型名、鉴权头。
- 是否支持 `response_format`。
- JSON 返回不稳定时的 repair 和 fallback。
- CORS / 本地代理提示。
- 请求失败时的中文错误提示。

Provider 适配不应该写死某个供应商的产品策略，也不应该让某个模型的坏习惯污染通用访谈流程。

## 下一阶段候选改动

1. 增加访谈记忆去重
   - 记录每轮已问问题主题。
   - 下一轮禁止重复问同一主题，除非用户主动要求展开。

2. 增加问题难度分级
   - beginner：业务场景、用户、页面、验收。
   - intermediate：数据来源、权限、部署目标。
   - advanced：技术栈、架构、扩展性、成本。

3. 增加“交给 Agent 推荐”选项
   - 当用户不懂技术选择时，允许标记为“交给 Agent 推荐”。
   - ProjectSpec 中写入 `UNCONFIRMED`、`assumptions` 或 `recommendations`。

4. 优化 ProjectSpec 生成策略
   - 对用户没回答的技术细节，不强行追问。
   - 生成文件时明确哪些是用户确认事实，哪些是 Agent 建议。

5. 增加真实访谈样例集
   - 电商小工具
   - Chrome 插件
   - 独立站
   - 本地效率工具
   - AI 工作流工具

6. 增加跨 Provider 回归样例
   - 同一份项目输入分别用不同 Provider 跑一遍。
   - 比较问题重复率、专业术语密度、JSON 合法率和最终 Markdown 可执行性。

## 不做的事

- 不把项目改成 SaaS。
- 不强制用户回答所有技术问题。
- 不把模型建议伪装成用户确认事实。
- 不要求代码小白理解数据库、部署和架构细节后才能生成任务包。
- 不把任何真实 API Key 写入文档、日志、示例或测试。

## 验收标准

- 代码小白完成 2-3 轮访谈后，可以得到可读 ProjectSpec。
- 重复问题明显减少。
- “后续再考虑”会进入待确认项，而不是继续追问。
- Markdown 文件中清楚区分用户确认内容和 Agent 建议。
- 至少 2 个不同 Provider 可以跑通同一套 Web 流程。
